# 04 · 价值主张、自治分级与发展路径

> 回答三个根本问题:这项目和直接用 Claude 差在哪?能不能真落地?能减少人工到什么程度?
> 这是项目的"为什么存在"和"卖点边界",简历话术从这里推导。

## 一、核心命题:你的项目 vs 直接用 Claude

**生死线**:若项目只是"把告警丢给 LLM 出根因",那就是 prompt 包装器,和把告警贴给 Claude 没区别,且更差(Claude 更强)。这种项目没有护城河,面试三句问穿。

**可落地的价值,必须在 LLM 做不到的地方。LLM 是大脑,工程是它的手脚/感官/记忆/约束:**

| 维度 | Claude 单独做不到 | 本项目提供 | 性质 |
|---|---|---|---|
| 感官(数据接入) | 看不见你的日志/指标/链路 | Agent 主动 tool calling 查询 | 工程 |
| 手脚(执行) | 不能重启/扩容/回滚 | 动作执行端(Mock) | 工程 |
| 约束(可靠性) | 会幻觉、乱调工具、输出不稳 | 确定性编排 + schema 校验 + 安全闸 | 工程,核心 |
| 记忆(经验) | 每次无状态,忘光 | 案例库 + 服务画像,越用越准 | 数据+工程,最大差异化 |
| 自动化(规模) | 没人半夜贴告警给它 | 7×24 自动接入→诊断→处置 | 工程 |

> **一句话**:LLM 是组件不是产品。LLM 只出现在 2 个叶子节点(根因推理、修复建议),其余全是确定性工程。"可落地"= 用工程把不可靠的 LLM 包在中间,外面全是可测可控可审计的代码。

## 二、自治分级 —— "减少人工"的正解

**目标(用户提出)**:尽量减少人在诊断-处置中的参与,理想是 AI 自动查日志→判断→处置→重启。

**红线**:`AI 自动改生产代码并重启` = 核按钮,**不做**。风险不对称:
- 诊断错 → 浪费几秒,无害
- 自动改代码错 → 单服务故障扩散成集群雪崩(Never break userspace)
- 现实中没有公司允许 AI 直接改生产代码;简历这么写=面试官判定"不懂生产",减分

**正确做法:把"人工审批"换成"按爆炸半径分级的自动化策略门(policy gate)"**

| 动作 | 爆炸半径 | 自治级别 |
|---|---|---|
| 查日志/指标/链路 | 零(只读) | 全自动 |
| 清缓存 / 限流 / 降级开关 | 单服务,秒级可回滚 | 自动执行 + 事后通知 |
| 重启实例 / 扩容 | 单服务,可观测 | 自动执行 + 熔断(连续 N 次停手报人) |
| 回滚到上个版本 | 中等,回到已知安全态 | 自动执行(比改代码安全得多) |
| 改代码 / 改配置 / 改数据 | 大,不可控 | **永远走 PR + CI,人 review,不直接上** |

**关键洞察**:90% 线上故障靠重启/回滚/扩容/限流就能解决,**根本不需要改代码**。改代码是最差的自愈手段(慢、危险、需测试)。成熟自愈系统恰恰回避改代码。

**修正后的闭环:**
```
故障 → AI 诊断(全自动)→ 判断动作类型
   ├─ 低危(重启/限流/扩容/回滚)→ 自动执行 → 金丝雀验证 → 通知
   └─ 改代码 → 生成 PR + 跑 CI → 人 review PR(不自动 merge)
```
人没从"每次诊断"退出,而是从"低危动作"退出,只在"改代码"高危处保留。这才是真减人 + 不搞挂生产。

## 三、让"无人"安全的五道闸 —— 真技术点在这

敢不敢"无人",不看 AI 多聪明,看这五道闸够不够硬。这是和"prompt 包装器"及"鲁莽玩具"的全部差距:

1. **置信度门控**:AI 给根因打置信分,> 阈值才自动执行,否则降级报人
2. **动作白名单 + 幂等**:AI 只能从预定义安全动作集选,不能自创;同动作重复执行无副作用
3. **熔断器**:同服务 5 分钟内自愈动作 ≥ N 次 → 判定"治不好",停手升级给人(防 AI 在错误判断上反复横跳)
4. **金丝雀 + 自动回滚**:执行后盯指标 N 分钟,未恢复或更糟 → 自动回滚(闭环的闭环)
5. **全审计**:每个自动动作留痕(哪个 Agent / 根因+置信度 / 做了什么 / 结果),可追溯

## 四、Memory 闭环 —— 最大差异化,把自治和 RAG 串起来

RAG = 被动检索静态知识;Memory = 主动积累动态经验。结合 = 系统会学习。

- **短期记忆**:诊断会话内工具结果累积(`DiagnosticContext` 即雏形)
- **长期记忆**(真讲头):
  - **案例库自增长**:每次成功处置(或人工确认)的诊断,固化成新 RAG 文档,越用越准
  - **实体记忆**:记住每个服务的历史故障模式 + 基线指标,告警时带画像诊断而非冷启动

Memory 技术深度在四个决策:
1. **记什么**:不全存,只提炼(确认的根因、服务画像)
2. **怎么检索**:语义相似 + 实体匹配(serviceName)+ 时间衰减
3. **怎么固化**:成功处置/人工确认后才写入,防错误经验污染
4. **怎么遗忘**:旧故障模式失效要过期/降权

**Memory 驱动自治进化(逐步减人的正解):**
```
新故障模式 → 谨慎,要人确认动作
同模式处理 10 次,成功率高 → memory 记录
第 11 次同模式 → 查 memory:"处理过 10 次,重启成功率 100%" → 自动降级审批门,直接执行
```
不是一刀切移除人工,是**系统用历史经验把验证过的安全动作自动提权**。人只在新场景/低置信/高危处介入,系统越用人越省。这条线把 memory + 自治分级 + RAG 案例库三者串成一个自洽故事。

## 五、发展路径(按差异化 × 工作量排序)

| 路径 | 内容 | 评价 |
|---|---|---|
| A · 单点做深 | 六个 Phase 别平均用力,挑收敛或 RAG 质量做出真数据 | 一个深点 > 六个浅点 |
| B · 自愈闭环 | 诊断→动作状态机(PROPOSED→APPROVED→EXECUTING→VERIFIED/ROLLED_BACK)+ 分级自治。执行端 Mock 但状态机真。复用现有 FixAgent 经验 | 差异化大,**推荐** |
| C · 统一基础设施 | 旧代码审查 Agent + 新诊断 Agent **共享基础设施层**(LLM 客户端/SSE/RAG/Agent 接口/虚拟线程)。**注意边界:卖"一套基础设施支撑两个垂直应用",不卖"通用 Agent 大脑"(那是过度抽象)** | 把旧代码从负担变资产,**推荐** |
| D · 评估可观测 | 根因准确率离线评测集、Agent trace 可视化、token 成本、输出质量打分。已有 `rag-evaluation-dataset.csv` 可延续 | 能证明 Agent 有效的人稀缺,真区分度 |

**推荐组合:C(基础设施复用)+ B(自愈状态机 + 分级自治)+ Memory 闭环。** 全建立在已有资产上(FixAgent 复用、RAG 评估数据、双 Agent 共享框架),不从零堆功能,叙事最硬。

## 六、能讲深的技术点(替代烂大街的"我用了 RAG/多 Agent")

1. **Tool calling 可靠性工程**:LLM 会调不存在的工具/传错参/幻觉 → schema 强校验、参数白名单、失败确定性兜底、重试降级。Agent 系统最硬的真问题
2. **确定性 vs 自主性权衡**:Supervisor 为何用确定性并行而非 LLM 调度(可测性取舍)
3. **分级自治 + 五道安全闸**:见 §二、§三
4. **Memory 闭环 + 自治提权**:见 §四
5. **告警降噪**:不止 SET NX EX,相似告警聚类成事件 + 抑制率指标
6. **RAG 结构化切分 + 评估**:按"现象/排查/根因/修复"语义切;召回率/精确率评测(延续已有数据集)
7. **流式背压与错误恢复**:SSE 中途 LLM 超时的优雅降级

## 七、简历话术(诚实版,基于本方案)

**项目描述**
> 基于 Java 21 + Spring Boot 3 WebFlux + LangChain4j 的微服务 AIOps 自愈平台:告警接入→收敛→多 Agent 根因诊断→RAG 案例检索→分级自治处置→SSE 流式输出,带 memory 经验沉淀闭环。

**可写亮点**
- 多 Agent Supervisor 协同 + Reactor 并行根因诊断
- 分级自治处置:按动作爆炸半径自动/人工分流,置信度门控 + 熔断 + 金丝雀自动回滚五道安全闸
- Memory 闭环:成功处置经验固化为案例库 + 服务故障画像,驱动审批门自动提权(系统越用越自治)
- RAG 运维知识库:结构化语义切分 + 向量检索 + 召回质量评测
- Tool calling 可靠性工程:schema 校验 + 参数白名单 + 失败兜底
- Redis 滑动窗口告警收敛 + 聚类降噪
- WebFlux SSE 流式诊断 + Java 21 虚拟线程承载阻塞查询
- MongoDB 诊断记录持久化 + 全动作审计

**❌ 别写(会被问穿)**
- AI 自动修改生产代码并重启(危险,不做;改代码走 PR+CI)
- 真实接入 Prometheus/ELK/SkyWalking(Mock)
- RabbitMQ 高吞吐 / BM25+Rerank(已砍)
- 高可用集群(单机 demo)

**答辩口径**
- "为什么不让 AI 自动改代码":风险不对称,改代码走 PR+CI;自愈优先用可回滚的运维动作(重启/回滚/扩容/限流),覆盖 90% 场景
- "怎么敢自动执行":置信度门控 + 动作白名单 + 熔断 + 金丝雀自动回滚 + 全审计五道闸
- "和直接用 Claude 区别":LLM 是组件,价值在感官/手脚/记忆/约束/自动化(见 §一)
- "外部系统 Mock 是缺陷吗":刻意工程取舍,接口已隔离,换真实数据源不动业务
